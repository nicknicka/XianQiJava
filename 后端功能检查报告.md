# 后端聊天功能检查报告

## 📊 数据库设计分析

### 1. 会话表 (conversation)

**设计模式**: **两个用户共用一个会话记录**

```sql
CREATE TABLE conversation (
  conversation_id BIGINT PRIMARY KEY AUTO_INCREMENT,
  conversation_type INT COMMENT '会话类型：1-单聊，2-群聊',
  user_id_1 BIGINT COMMENT '用户1ID',
  user_id_2 BIGINT COMMENT '用户2ID',
  related_product_id BIGINT COMMENT '关联商品ID',
  related_order_id BIGINT COMMENT '关联订单ID',
  last_message_id BIGINT COMMENT '最后一条消息ID',
  last_message_content VARCHAR(500) COMMENT '最后一条消息内容',
  last_message_time DATETIME COMMENT '最后消息时间',

  -- 用户1的个性化设置
  unread_count_user1 INT DEFAULT 0 COMMENT '用户1的未读消息数',
  is_muted_user1 TINYINT DEFAULT 0 COMMENT '用户1是否免打扰',
  is_archived_user1 TINYINT DEFAULT 0 COMMENT '用户1是否归档',
  is_pinned_user1 TINYINT DEFAULT 0 COMMENT '用户1是否置顶',
  pin_order_user1 INT COMMENT '用户1的置顶排序',
  remark_user1 VARCHAR(100) COMMENT '用户1对会话的备注名',

  -- 用户2的个性化设置
  unread_count_user2 INT DEFAULT 0 COMMENT '用户2的未读消息数',
  is_muted_user2 TINYINT DEFAULT 0 COMMENT '用户2是否免打扰',
  is_archived_user2 TINYINT DEFAULT 0 COMMENT '用户2是否归档',
  is_pinned_user2 TINYINT DEFAULT 0 COMMENT '用户2是否置顶',
  pin_order_user2 INT COMMENT '用户2的置顶排序',
  remark_user2 VARCHAR(100) COMMENT '用户2对会话的备注名',

  status INT DEFAULT 0 COMMENT '状态：0-正常，1-删除',
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
  update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

  UNIQUE KEY uk_users (user_id_1, user_id_2, related_product_id)
);
```

**特点**：
- ✅ 单个会话记录支持两个用户
- ✅ 每个用户有独立的设置（免打扰、置顶、归档等）
- ✅ 支持基于同一对用户创建多个会话（通过related_product_id区分）
- ✅ 通过user_id_1 < user_id_2保证唯一性

### 2. 消息表 (message)

**设计模式**: **两个用户共用消息记录**

```sql
CREATE TABLE message (
  message_id BIGINT PRIMARY KEY AUTO_INCREMENT,
  conversation_id BIGINT COMMENT '会话ID',
  from_user_id BIGINT COMMENT '发送者ID',
  to_user_id BIGINT COMMENT '接收者ID',
  content TEXT COMMENT '消息内容',
  type INT COMMENT '类型：1-文本，2-图片，3-商品卡片，4-订单卡片，5-系统通知，6-引用消息',
  parent_message_id BIGINT COMMENT '引用的父消息ID',
  is_read TINYINT DEFAULT 0 COMMENT '是否已读：0-未读，1-已读',
  read_time DATETIME COMMENT '阅读时间',
  send_status TINYINT DEFAULT 0 COMMENT '发送状态：0-发送中，1-成功，2-失败',
  delivered_time DATETIME COMMENT '送达时间',
  extra_data JSON COMMENT '扩展数据',
  reply_count INT DEFAULT 0 COMMENT '被回复次数',
  status INT DEFAULT 0 COMMENT '状态：0-正常，1-撤回，2-删除',
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
  update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

  INDEX idx_conversation (conversation_id),
  INDEX idx_from_user (from_user_id),
  INDEX idx_to_user (to_user_id)
);
```

**特点**：
- ✅ 一条消息记录同时属于两个用户
- ✅ 通过conversationId关联会话
- ✅ fromUserId和toUserId明确发送方和接收方
- ✅ 支持消息撤回（status=1）和删除（status=2）

## 🚨 需要修复的问题

### 1. 缺少清空聊天记录接口

**问题描述**:
- 前端已调用 `DELETE /conversation/{id}/messages`
- 后端Controller中没有实现这个接口

**影响**:
- 用户点击"清空聊天记录"会报错
- 无法清空历史消息

**修复方案**: 在ConversationController中添加接口

```java
/**
 * 清空聊天记录
 */
@Operation(summary = "清空聊天记录")
@DeleteMapping("/{id}/messages")
public Result<Void> clearMessages(
        @Parameter(description = "会话ID") @PathVariable("id") Long id) {
    Long userId = SecurityUtil.getCurrentUserIdRequired();
    log.info("清空聊天记录, conversationId={}, userId={}", id, userId);
    conversationService.clearMessages(id, userId);
    return Result.success("聊天记录已清空");
}
```

### 2. ConversationService接口缺少clearMessages方法

**修复方案**: 在ConversationService接口中添加方法定义

```java
/**
 * 清空聊天记录
 *
 * @param conversationId 会话ID
 * @param userId        当前用户ID
 */
void clearMessages(Long conversationId, Long userId);
```

## 💡 消息存储设计说明

### 问题：是一个用户有一个记录还是两个用户共用一个记录？

**答案**: **两个用户共用一个会话记录和消息记录**

### 设计细节

#### 1. 会话存储方式

**共享设计**：
```
User A (ID: 1) ←→ Conversation ←→ User B (ID: 2)
                      (ID: 100)
```

- User A 和 User B 之间只有**一个**会话记录（conversationId = 100）
- 两个用户在同一个会话中看到相同的消息
- 每个用户有独立的个性化设置：
  - 未读数（unreadCountUser1 vs unreadCountUser2）
  - 置顶状态（isPinnedUser1 vs isPinnedUser2）
  - 免打扰设置（isMutedUser1 vs isMutedUser2）
  - 归档状态（isArchivedUser1 vs isArchivedUser2）

**多会话支持**：
- 同一对用户可以基于不同商品创建多个会话
- 通过 `related_product_id` 区分不同的会话
- 例如：
  - 会话1：用户A和用户B关于商品X的讨论
  - 会话2：用户A和用户B关于商品Y的讨论

#### 2. 消息存储方式

**共享设计**：
```
Message (ID: 1001)
  ├─ conversationId: 100
  ├─ fromUserId: 1 (User A)
  ├─ toUserId: 2 (User B)
  ├─ content: "你好"
  └─ status: 0 (正常)
```

- 一条消息记录同时属于两个用户
- 两个用户都能看到这条消息
- 通过 `fromUserId` 和 `toUserId` 区分发送方和接收方
- 消息撤回/删除是全局的（一旦撤回，双方都看不到）

#### 3. 优缺点分析

**优点**：
- ✅ 节省存储空间：不需要为每个用户复制一份消息
- ✅ 数据一致性：两个用户看到相同的消息历史
- ✅ 实现简单：不需要维护复杂的消息关联关系
- ✅ 性能较好：查询时只需要读取一条消息记录

**缺点**：
- ❌ 缺乏隐私性：无法实现"单方面删除消息"功能
- ❌ 撤回限制：一旦撤回，双方都看不到
- ❌ 删除限制：无法只删除自己这边的消息

**适合场景**：
- ✅ 校园二手交易应用（信任环境）
- ✅ 需要完整聊天记录保存的场景
- ✅ 消息审计和纠纷处理

**不适合场景**：
- ❌ 需要强隐私保护的应用
- ❌ 需要单方面删除消息的功能

## 🔧 后端实现示例

### ConversationServiceImpl - 清空消息实现

```java
@Override
@Transactional(rollbackFor = Exception.class)
public void clearMessages(Long conversationId, Long userId) {
    log.info("清空聊天记录, conversationId={}, userId={}", conversationId, userId);

    // 1. 验证会话是否存在且用户有权操作
    Conversation conversation = this.getById(conversationId);
    if (conversation == null || conversation.getStatus() == 1) {
        throw new BusinessException("会话不存在");
    }

    if (!conversation.getUserId1().equals(userId) &&
        !conversation.getUserId2().equals(userId)) {
        throw new BusinessException("无权操作此会话");
    }

    // 2. 逻辑删除所有消息（设置status=2）
    LambdaUpdateWrapper<Message> updateWrapper = new LambdaUpdateWrapper<>();
    updateWrapper.eq(Message::getConversationId, conversationId)
                 .set(Message::getStatus, 2);
    messageService.update(updateWrapper);

    log.info("清空聊天记录成功, conversationId={}, affectedRows={}",
             conversationId, updateWrapper.getCount());

    // 3. 更新会话的最后消息信息
    conversation.setLastMessageId(null);
    conversation.setLastMessageContent(null);
    conversation.setLastMessageTime(null);
    this.updateById(conversation);

    // 4. 重置未读数
    if (conversation.getUserId1().equals(userId)) {
        conversation.setUnreadCountUser1(0);
    }
    if (conversation.getUserId2().equals(userId)) {
        conversation.setUnreadCountUser2(0);
    }
    this.updateById(conversation);
}
```

## 📝 总结

### 消息存储设计

**设计模式**: 两个用户共用一个会话记录和消息记录

**核心特点**:
- 会话表：一个会话包含两个用户，每个用户有独立设置
- 消息表：一条消息同时属于两个用户，通过fromUserId/toUserId区分角色

**优势**:
- 节省存储空间
- 保证数据一致性
- 实现简单高效

### 缺失功能

**必须实现**: 清空聊天记录接口
- Controller层：添加DELETE接口
- Service接口：添加方法定义
- ServiceImpl：实现清空逻辑

### 建议

1. **立即修复**: 添加清空聊天记录功能
2. **优化体验**: 支持逻辑删除而非物理删除
3. **长期规划**: 考虑添加消息已读回执和转发功能
